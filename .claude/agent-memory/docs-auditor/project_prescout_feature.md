Prescout data and photo gallery were added as non-config-driven features in March 2026. In April 2026, the shared `prescout_data` and `team_photos` tables were replaced with per-game tables following the existing scouting table pattern.

**DB tables (per-game, created in `createGame()`, dropped in `deleteGame()`):**
- `prescout_<gameName>` — `team_number UNIQUE`, `data` JSONB; no `game_name` column; isolation by table name
- `photos_<gameName>` — `team_number`, `photo_data TEXT`, `mime_type`, `uploaded_by`, `uploaded_at`; no `game_name` column

Table names generated by `sanitizePrescoutTableName()` and `sanitizePhotosTableName()` in `schema-generator.js`.

**Key behavioral nuances:**
- `photos/[id]` GET and DELETE routes now require `?gameId=<id>` query param to resolve which per-game table to use.
- `PhotoGallery.js` component accepts a `gameId` prop for per-game photo route resolution.
- Re-upload upserts on `UNIQUE(team_number)` — no `game_name` in the unique constraint.
- Photos have no access control beyond "any authenticated user" — any scout can upload or delete any photo.
- Spreadsheet must have a sheet named exactly `"Prescout"` (case-sensitive). Transposed layout: row 0 = team numbers, col 0 = field names.
- `xlsx` (SheetJS) package used for server-side parse.

**Page integration:** `/team-view` and `/compare` show PrescoutSection + PhotoGallery (read-only); `/scout-leads` shows PhotoGallery with upload/delete; `/admin/prescout` is the upload/clear admin UI.

**Why non-config-driven:** Prescout data shape varies per event and is imported externally, not collected via the scouting form.

**How to apply:** When reviewing or updating docs, note that prescout/photo are app-level features independent of the game config JSON. Document them separately from the config reference sections. The old shared-table architecture (`prescout_data`, `team_photos`) is gone — do not reference it.
